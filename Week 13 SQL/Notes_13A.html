Context

  • A relational database (or RDB) is a structure for organizing information that one may think of initially as a supercharged version of a data frame. (The word â€œrelationalâ€ does not actually indicate that relationships can exist between separate units in your database. It actually has to do with the concept of relational algebra, first proposed in this paper by E. F. Codd.)

  • In an RDB, the basic unit of data storage is the table (or relation or schema). Rows are rows, but sometimes dubbed tuples or records, and columns are sometimes dubbed attributes. (Fields are specific cells within a given row.)

  • RDBs come with performance guarantees, dubbed ACID.

  • When designing an RDB, a key goal is to eliminate redundancy by having many tables that are linked via foreign keys. For instance, if you want to construct an RDB having information about movies, actors, and studios (where, e.g., a movie is produced by a studio and stars an actor), it makes the most sense to split the information cleanly between interlinked tables.

  • RDBs are commonly manipulated and queried using SQL: Structured Query Language.

Preliminary Concepts I: Tables

We note the following about tables:

  • each table in a database represents a single entity;

  • each row of a table represents one instance of that entity;

  • each column of a table must have a name;

  • each column of a table must have a pre-specified data type; and

  • the columns in each row must satisfy any constraints specified with the table is created.

Preliminary Concepts II: Data Types

As stated on the last slide, the type of each data must be specified as a table is created! In contrast to R, Postgres offers a wide variety of (sometimes very specific) data types, such as

  • numeric: arbitrary precision numbers

  • serial: auto-incrementing integer

  • text: variable, unlimited-length character string

  • money: currency values

  • date: a specific day

  • timestamp: a specific moment, day and time (to the nanosecond)

  • interval: a time interval

  • boolean: true or false

See the data type documentation page for more details. (And there are many.)

Preliminary Concepts III: Keys

A unique key may be defined as one (or more) columns whose collective values uniquely identify each row of a table.

If a table has two or more unique keys, it is customary to define one of them as the primary key.

When we link tables, a foreign key in one table is a reference to the primary key in the other.
